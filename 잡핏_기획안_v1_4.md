## 1. 서비스 개요

잡핏(JobFit)은 웨딩홀 서빙·택배 물류 등 단기·반복적 알바를 전문으로 하는 **인력 알선 플랫폼**입니다. 의뢰 기업으로부터 인력 수급 요청을 받아 알바생을 연결하며, 사장님(테스트마스터) 1인이 직접 운영하는 단일 사업자 구조입니다.

### 1-1. 비즈니스 구조 — 잡핏은 중계업체

| 구분 | 누가 지급 | 금액 | 코드 필드 |
|---|---|---|---|
| **알바비** (일급/시급) | 파트너사 → 알바생 직접 | 100,000~125,000원 | `j.wage` (단위: **원**) |
| **잡핏 포인트** | 잡핏 → 알바생 | 기본 1,000P + 보너스 | `pointRewardFor(j)` (단위: **P**) |

→ 잡핏은 **포인트만 지급**, 알바비 지급 책임은 파트너사. 정산·송금 자료는 정산 리포트로 안내만 제공.

### 1-2. 플랫폼 구성

| 플랫폼 | 대상 | 주요 기능 | 현재 상태 |
|---|---|---|---|
| **관리자 웹페이지** | 마스터 + 관리자 1·2등급 | 공고·승인·근무자·포인트·관제·통근버스·정산 모두 | ✅ 프로토타입 완성 (디자인 레퍼런스) |
| **관제 시스템 (웹)** | 모든 관리자 | 실시간 출근 현황 통합 대시보드 | ✅ 관리자 웹 통합 |
| **알바생 앱** (iOS/Android) | 알바생 | 공고 검색·신청·출근·포인트 관리 | ⏳ 관리자 웹 연동 후 제작 |
| **관리자 앱** (iOS/Android) | 마스터·1·2등급 | 현장에서 승인·출근 확인·문의 응대 | ⏳ 관리자 웹 연동 후 제작 |

## 2. 출시 로드맵 (v1.4 신규 — Phase 분리)

### Phase 1 — 관리자 웹 프로토타입 ✅ **완료**

- 18개 페이지 + 모달 다수 (디자인 + 비즈니스 로직)
- 11개 데이터 entity 모델링 + Supabase 스키마 가이드
- mutation 헬퍼 30+ 개 (Supabase RPC 매핑 가능)
- 권한 3단계 + 정책 상수 통합 (`POLICY` 객체)
- 통근버스 매트릭스 모델 + 자동/수동 알림 시스템

→ **외부 개발자 인계 직전 상태**. 디자인은 직원 내부용이라 폴리시 생략.

### Phase 2 — Supabase 연동 (예상 2~3개월)

1. **Supabase 프로젝트 셋업** (Auth/Storage/RLS)
2. 11+ 테이블 스키마 적용 ([HANDOFF.md §3](HANDOFF.md))
3. 시드 데이터 → SQL INSERT 마이그레이션
4. mutation 헬퍼 → Supabase RPC/Edge Functions 매핑
5. 관리자 웹을 Next.js/Vite + React로 재구현 (현재 IIFE 코드 = 디자인·플로우 참고용)
6. RLS 정책 적용 (3단계 권한 × 11테이블)
7. 실 출결 테이블 도입 (현재 `getAttendance` 시뮬 제거)
8. FCM 프로젝트 + 카카오 비즈니스 채널 연결
9. 운영 환경 배포 (Vercel/Cloudflare + Supabase Pro)

### Phase 3 — 알바생 앱 + 관리자 앱 (예상 3~4개월)

- **Flutter 3.x** (iOS/Android 단일 코드베이스)
- 카카오 로그인 + GPS 권한 + FCM 푸시 + 카카오맵
- 알바생 앱 5+ 화면: 홈 / 알바찾기 / 내근무 / 포인트 / 1:1문의 / 같이하기 / 통근버스 가이드
- 관리자 앱 (현장용): 4탭 (홈 / 출결 / 승인 / 경고)
- 베타 테스트 (한 근무지 1주 시범) → 정식 출시

### Phase 4 — 정식 출시 (예상 1개월)

- 전 11개 근무지 확장
- 마케팅 (긴급 구인 알림 활성)
- 모니터링 인프라 (Sentry, Grafana)

→ **현실적 출시 시점: 6~9개월 후** (사장님 결정 + 외주 인력 1~2명 동원 시)

## 4. 핵심 기능 상세

### 4-1. 회원가입 (알바생)

- 카카오 계정 간편 가입
- 기본 정보: 이름 / 생년월일 / 휴대폰 / 성별 / 지역
- 급여 계좌: 은행 + 계좌번호 + 예금주
- 개인정보 활용 동의 + 안전교육 확인 (서명)
- 선호 업종/지역 우선순위 리스트
- 마케팅 / 긴급 구인 / 야간 알림 수신 동의 (각각 선택)
- 미성년자(18세 미만) 가입 불가
- 탈퇴 후 30일간 재가입 불가 (전화번호 기준) — Phase 2 정책

### 4-2. 출근 시스템 — GPS 다각형 영역 검증

근무지별 영역(폴리곤) 기반 검증 (500m 고정 반경 폐지).

- 근무 1시간 전 출근 버튼 활성화 (영역 안에서만 클릭 가능)
- 출근 버튼 → 확인 팝업 → 최종 확인 (2단계)
- 활성화된 계약서/안전교육 자료에 서명
- GPS 영역 재검증 → 출근 상태로 변경
- 관리자 앱/웹 출근 명단에 실시간 반영

| GPS 상태 | 표시 | 색상 |
|---|---|---|
| 영역 내 + 시작시간 ~ 시작시간 | 출근 확인 | 🟢 녹색 |
| 영역 내 + 시작시간+30분 사이 | 지각 | 🟡 노란색 |
| 시작+30분 미확인 | 결근 | 🔴 빨간색 |

### 4-3. 퇴근 시스템

| 상황 | 처리 방식 |
|---|---|
| 영역 내 | 퇴근 + GPS 확인 시 포인트 자동 지급 |
| 영역 밖 | 퇴근 + 사유 입력 → 관리자 승인 → 포인트 지급 |
| 종료 시간 도달 | 푸시 알람 + 5분 단위 반복 알람 |
| 종료 + 6h 후 | 자동 퇴근 처리 |

### 4-4. 자동 알림 (v1.4 신규)

**근무 시작 1시간 전 신청자 자동 푸시**

- `POLICY.PRE_WORK_REMINDER_MIN = 60` (분, 정책 상수)
- 대상: 해당 공고 신청자 중 approved/pending/cancel_pending
- 공고별 **on/off 토글** (기본 ON, 마스터·1·2급 모두 토글 가능)
- 발송 시각 / 발송자(`시스템(자동)`) / 본문 / 수신자 ID 풀 저장
- 토스트로 운영자에게도 발송 시점 안내
- 본문 자동 생성 (통근버스 운영 공고면 추천 차수 안내 포함)

### 4-5. 신청자 알림 (v1.4 신규)

**공고별 자유 안내 — 폭설/사고/지연 등**

- 공고 상세 [📢 신청자 알림] 버튼
- 마스터·1·2급 모두 발송 가능
- 빠른 템플릿 5개 (폭설/사고/결행/일반/근무변경)
- 자유 입력: 제목 60자 + 본문 500자
- 🚨 긴급 토글 (야간 22~08 제한 우회)
- 푸시 미리보기 + 대상자 칩 리스트
- **본문 + 수신자 ID + 발송 시각 + 발송자 풀 저장** (Supabase 전환 시 그대로 매핑)

### 4-6. 긴급 구인 알림

- 마스터/관리자가 근무지 단위로 발송
- 회원가입 시 **긴급 구인 수신 동의자에게만** 발송
- 수락 알바생에게 추가 포인트 지급
- 긴급 공고는 알바 찾기 최상단 고정

### 4-7. 경고 및 협의대상 시스템

#### 4-7-1. 경고 부여 5사유 (관리자 재량)

| 사유 | 관리자 판단 |
|---|---|
| 12시간 이내 취소 (사유 작성 필수) | 패스 또는 경고 1회 |
| 근무 지각 | 패스 또는 경고 1회 |
| 무단결근 (시작+30분 GPS 미확인) | 패스 또는 경고 1회 |
| 무응답 / 연락두절 | 패스 또는 경고 1회 |
| GPS 미검증 결근 | 패스 또는 경고 1회 |

→ 자동 부여가 아닌 **관리자 재량 판단**. 경고 부여 시 사유 필수 + 알바생 알림.

#### 4-7-2. 경고 누적 → 자동 협의대상

- 1회: 홈 화면 뱃지
- 2회: "다음 경고 시 협의대상 등록" 안내
- 3회: **자동 협의대상 등록** (자동 승인 차단)

#### 4-7-3. 협의대상 정책 (전화번호 기반)

- 협의대상 = 전화번호 기반 관리
- 신규 가입 시 전화번호 자동 매칭 → 즉시 협의대상 등록
- **해제는 마스터 전용** (영구 차단 없음)
- 알바생은 자기가 협의대상인지 확인 불가

### 4-8. 취소 정책 (v1.4 6사유 카테고리)

#### 4-8-1. 취소 시점

| 시점 | 처리 |
|---|---|
| 근무 시작 12시간 전까지 | 자유 취소 (패널티 없음) |
| 12시간 이내 | 사유 카테고리 선택 → 관리자 검토 |
| 단순 변심 | -1,000P 자동 차감 |

#### 4-8-2. 사유 카테고리 6종 + 권고 액션

| 카테고리 | 권고 |
|---|---|
| `normal` 단순 변심 | **차감** (-1,000P) |
| `sick` 본인 질병 | **면제** (증빙 권장) |
| `family` 가족 응급 | **면제** |
| `transport` 대중교통 장애 | **면제** |
| `weather` 천재지변 | **면제** |
| `other` 기타 | **메모 확인 후 결정** |

#### 4-8-3. 관리자 3액션

- **차감 승인** (`approveCancelDeduct`) → -1,000P + 슬롯 회수
- **면제 승인** (`approveCancelExempt`) → 차감 없이 슬롯 회수
- **반려** (`rejectCancelRequest`) → 신청 복원 (출근 의무)

### 4-9. 같이하기 (Buddy / 친구 호출) 정책 v1

알바생 1명이 친구 1명을 초대해 같은 공고에 페어로 신청. 양쪽 출퇴근 완료 시 **각자 +3,000P**.

#### 핵심 정책 8개

| # | 정책 | 결정 |
|---|---|---|
| 1 | 자리 부족 시 (<2자리) | 둘 다 신청 불가 + 알바생 앱 안내 |
| 2 | 한쪽 협의대상/경고3회 | 둘 다 관리자 검토 |
| 3 | 한쪽 자유 취소 (12h 전) | 짝꿎 알림 + 보너스 자격 양쪽 소멸 |
| 4 | 결근/지각 보너스 | 양쪽 정시 출근만 자동 / 정정 시 수동 |
| 5 | 친구 응답 없음 | 10분 자동 취소 |
| 6 | cascade reject 후 재신청 | 자리 여유 있을 때만 단독 가능 |
| 7 | 포인트 합산 | 공고 포인트 + 같이하기 3,000P 별도 |
| 8 | 그룹 크기 | 1:1 페어만 |

### 4-10. 1:1 문의 채팅 (v1.4 재구성)

기존 단순 message/answer → **카카오 고객센터 패턴** 하이브리드:

- 게시판 (전체 문의 리스트) + 1:1 스레드 (채팅)
- 알바생 새 문의 → `pending`
- 운영자 답변 → `in_progress`
- 알바생 추가 메시지 → `pending` (응답 우선순위 ↑)
- 운영자 종결 → `closed`
- 종결 후 알바생 메시지 → 자동 `in_progress` (재오픈)

### 4-11. 지각 보고 (v1.4 신규)

알바앱 [내 근무] 오늘 카드의 [⏰ 늦어요 보고] 버튼.

- 예상 지연 + 사유 입력
- 자동으로 1:1 문의 생성 + 🚨 긴급 마킹
- 관리자 [문의] 페이지에 즉시 표시 + 양방향 채팅 가능

### 5-2. 신청 승인 플로우

| 조건 | 처리 |
|---|---|
| 12시간 전 신청 | 자동 승인 (즉시) |
| 12시간 이내 신청 | 마스터/관리자 승인 필요 |
| 협의대상 / 경고 3회 | 시간 무관 무조건 관리자 승인 |
| 승인 대기 6h 초과 | 정책 미정 (이월 N19) |

### 5-3. 신청자 직접 추가 (v1.4 신규)

전화/카톡으로 신청 의사 받은 알바생 직접 등록:

- 공고 상세 [+ 알바생 직접 추가]
- 검색 → 자격 검증 (협의대상 / 주 4일 / 같은 날 다른 공고 / FULL)
- 절대 우회 불가: 이미 신청 / 같은날 / FULL / 데이터 오류
- 우회 가능: 협의대상 / 주 4일 초과 (force=true)

### 5-4. 관리자 강제 취소 (v1.4 신규)

알바생이 1:1 문의로 못 오는 사정 알린 후 운영자 처리:

- 신청자 목록 [신청 취소] 버튼 (open/closed 공고만)
- status='approved'만 처리 / `j.apply -= 1`
- buddy cascade 안 함 (짝꿎 승인 유지, 보너스 자격만 소멸)

### 5-5. 대기열 시스템

- FULL 시 대기 신청 (모집인원 × 2)
- 취소 발생 → 대기 1번에게 자리 제안 알림
- 수락 제한: 24h 전 1시간 / 24h 이내 30분
- 시간 초과 → 자동 거절 → 다음 대기자
- 모든 대기자 실패 → REOPENED (일반 모집 재개)

→ 관제 시스템은 등급 무관 전원 접근 (전 근무지 열람)

## 12. 통근버스 운영 중단 대응 (v1.4 신규 — 이월 N31)

### 12-1. 시나리오

1. **폭설/태풍** — 노선 전체(또는 일부) 운휴
2. **사고/지연** — 특정 차수 1회 운휴, 알바생 다른 차수로 재배치
3. **기사 결근** — 한 노선 1일 운휴
4. **차량 정비** — 일부 차수만 결행

### 12-2. 필요 데이터 모델 (Phase 2)

```sql
ALTER TABLE bus_routes ADD COLUMN status TEXT DEFAULT 'active'
  CHECK (status IN ('active','suspended','cancelled_today'));
ALTER TABLE bus_routes ADD COLUMN status_until TIMESTAMPTZ;
ALTER TABLE bus_routes ADD COLUMN status_reason TEXT;

CREATE TABLE bus_departure_cancellations (
  route_id        TEXT REFERENCES bus_routes(id),
  work_start_idx  INTEGER NOT NULL,
  cancelled_date  DATE NOT NULL,
  reason          TEXT,
  cancelled_by    TEXT,
  cancelled_at    TIMESTAMPTZ DEFAULT NOW(),
  PRIMARY KEY (route_id, work_start_idx, cancelled_date)
);
```

### 12-3. 사장님 결정 대기 정책 (이월 N31 세부)

- 통근버스 결행으로 알바생 지각/결근 시 페널티 면제 자동 적용 여부
- 차상위 노선 만석/시간 부족 시 택시비 지원 여부 + 한도
- 결행 결정 권한 — 마스터만 / 관리자 1급까지 / 현장 2급도 (담당 노선)
- 결행 알림 발송 시점 — 즉시 / 출발 30분 전 / 등록 시
- 결행 데이터 보관 기간 (통계 분석용)

### 12-4. 위험 운영 시나리오

- 한 알바생이 추천 차수 출발 직전(15분 내)에 결행 통보 → 환승 불가
- 야간 작업 마지막 차수 결행 → 알바생 출근 자체 불가 → 자동 취소 + 페널티 면제 + 보상 포인트?
- 폭설로 전 노선 결행 → 모든 신청 자동 취소 + 알바생/파트너사 동시 알림
